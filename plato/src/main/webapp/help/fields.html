Help Text for Input Fields

Descriptions for all input fields can be found here.

General

Field Description
XCDL XCL denotes the eXtensible Characterisation Languages, comprising a description language (XCDL) and an extraction language (XCEL). These languages describe the complete informational content of digital objects in abstract form to enable automated comparison of objects before and after migration in automated experiments.
There is extensive documentation about XCDL on the documentation page and the official XCL homepage.

Plan settings

Field Description
Private If a plan is set to private (which is the default), only the creator can see it.
Publish report For private plans, you can still make the uploaded report available to other users. They will not be able to change or open your plan, but see it in the list and download the report.
Uploading a report You can upload a report about your preservation plan and deposit it inside the plan. We recommend to do this in the following way:
  • Open your plan and go to "Validate plan" (or Analyse Results, if you havent progressed so far yet).
  • Expand everything, i.e. all the nodes in the trees that you want to see, and the evaluation and transformation section.
  • Print to PDF with the following settings: orientation landscape, two pages on one. This gives you a very compact layout in portrait with two landscape pages in small font. The DEMO plans provide such examples.
  • Upload the resulting PDF file here.

Define Basis

Field Description
Identification code Institutional unique identifier for the preservation plan
Document Types This is a a general description of the set of objects you are creating a preservation plan for.
Original technical environment The original set of software and hardware that was used to create and render the objects.
This description should be as detailed as possible to allow a solid comparison between environments and objects.
Plan Name Please provide a name for the preservation plan.
Plan Description This is a short description of the planning activity, basically a short summary.
Responsible Planner Responsible person for the planning activity. Please provide full name, job title, and contact.
Organisation Name of the organisation that is carrying out the planning activity.
Mandate Mandate of the repository (see TRAC A1.1)
Planning Purpose What is the purpose of the current planning activity?
Are you, for example, revising an existing plan because of changed environment factors or new objects?
Or are you defining a first plan for a set of objects that you have just received?
Designated community Identification of the designated community or communities and their associated knowledge base(s) (TRAC A3.1)
Applying policies Relevant polices for preservation planning activites (TRAC A3.3) - which policies are influencing your planning activity?
Relevant organisational procedures and workflows Relevant procedures, rules, manuals, handbooks and workflows for preservation planning endeavours (TRAC A3.3)
Contracts and agreements specifying preservation rights Relevant Contracts and deposit agreements of preservation rights (TRAC A5.2)
Reference to agreements of maintenance and access Relevant contracts and deposit agreements specifying all appropriate aspects of maintenance and access of the collection. (e.g. submission agreements, deposit agreements) This is especially relevant for data which are owned by third parties. Agreements may cover restrictions or requirements for access and maintenance, e.g. provide DIP in specific format or repository is responsible for error fixing.
Relations Are there any relations to other preservation plans? Is this plan for example based on another one replacing another plan, etc.?
Trigger Events that triggered this preservation planning activity (eg. new objects in the collection, a periodic review of the plan, changes in environment or requirements).
The following types of triggers are foreseen:
New collectionA plan is created for a new set of objects.
Periodic ReviewA scheduled review has led to the decision that an existing plan shall be revised.
Changed EnvironmentContextual influence factors have changed that justify a revision of a plan.
Changed ObjectiveThe goals of the institution have changed and require revising an existing plan.
Changed Collection ProfileThe profile of a set of objects has changed in such a way that current measures are not sufficient to keep the objects alive.
Policy File Upload You can upload your XML policy file here which will then be displayed and stored with the preservation project.

Define Sample Recods

Field Description
Collection ID Identification code of the collection, or a description that identifes it uniquely, if no code is available.
Description Description of the set of objects for which a preservation is created.
Type of objects What type of objects does the collection contain?
Number of Objects Number of objects currently in the considered collection.
Expected growth rate The expected growth rate of the collection (e.g. 100 objects per day).
Description of the sample records General description of the sample records and reasons for the specific selection of the sample records out of the whole set of objects.
Description Description of this specific sample record (What does it contain, why was it chosen, characteristics of the object,...)
Full Name Full name of the sample object.
Short Name Short name of the sample object.
PUID The Pronom Unique Identifier of the Sample Record. This can be determined by the DROID tool.
The PUID can be used for discovering applicable preservation actions that can operate on this object format in the later stages of evaluating potential actions (Define Alternatives).
Format Name The name of the format of the Sample Record.
Format Version The format version of the Sample Record.
Format Mime-Type The Mime-Type of the Sample Record.
Format default extension The common file extension of the Sample Record, such as 'jpg' for JPEG images.
File Upload You can upload a new sample object here using a normal file-upload.

Identify Requirements

Field Description
Description Additional information about the objective tree. What are the reasons for the chosen structure? Are there open issues at the moment? Are there specific reasons for not including particular influence factors, or for the ones that are included?
Attach files Additional information and documentation can be uploaded. This can for example be a report providing more in-depth coverage of the reasoning underlying the requirements tree specification.
Freemind tree upload You need to upload a valid Freemind XML File.
Please consult the corresponding help page for details.

Define Alternatives

Field Description
Name of Alternative Uniquely identifying short name of the potential strategy that is to be evaluated
Description of alternative The description should include required hardware, software and parameter settings.
Reason for considering Why was this specific alternative included in the evlauation procedure? Reasons might include previous experience with the selected tools, which might come from registries such as the Planets PA registry, constraints influencing the tool selection, etc.
Config Settings Configuration settings of tools and services of the alternative
Indicator of necessary resources Resources needed for executing and evaluating this alternative.

Go Decision

Field Description
Go Decision Select the decision taken about the continuation of the preservation planning process.
Reason for Decision State the reasons for your Go or No-Go decision here.
Action needed What further actions are needed depending on your Go/No-Go Decision?

Develop Experiment

Field Description
Develop experiment description Document the used hardware, software, parameters and the process to carry out the experiment as well as the mechanism to retrieve the results for each experiment.
Develop experiment: settings This field contains the actual parameter values used for calling the tool/service that you are evaluating. If you use an external tool yourself and specify the parameters manually, please document them here carefully.

Run Experiment

Field Description
Run experiment description Documentation about running the experiments for every sample object.
Report This text area contains a summary of the reports aggregated over all sample objects.
For automated migration services, usually the commandline output of the migration tools is automatically aggregated and displayed here. Emulators will have different logging output.
If a preservation action service fails with an error, this will be recorded here together with available information on the type of error, if applicable and available.
Resulting Files The resulting files after the experiments have been run. (eg. for migration the migrated files).
For migration web services that are executed by the planning tool, these files will be retrieved automatically. For all experiments that you are carrying out yourself, please upload all resulting files and documentation here.

Evaluate Experiment

Field Description
Comments Document any relevant observations and issues that you deem relevant for the evaluation process.

Transform Measured Values

Field Description
Comments Document any relevant observations and issues that you deem relevant for the transformation process.

Set Importance Factors

Field Description
Comments Document any relevant observations, issues and reasons for the weighting of importance factors that you have chosen.

Analyse Results

Field Description
Recommendation and reasoning The recommendation for one of the potential alternatives is based on an analysis of the strengths and weaknesses of the considered strategies. The reasons and observations should be recorded here to document the reasons for the decision. This is an essential part of the decision and should not be disregarded!
The recommendation forms the basis for specifying a preservation plan for the set of objects in the fourth phase of the planning process and thus needs to be thoroughly documented.

Create Executable Plan

Field Description
Triggers and conditions Enter here what triggers the execution and the conditions under which the preservation action will be carried out.
Validate and QA What actions have to be carried out to validate the preservation action and what documents have to be supplied to quality assure the execution.
Location of records In this field you should describe a path to the records. For automated execution of the plan this has to be entered in a for the execution tool understandable way.
Parameters for tool These are the parameters and settings of the execution tool. For automated execution of the plan this has to be entered in a for the execution tool understandable way.

Define Preservation Plan

Field Description
Currency The currency used for the cost model figures.
CIG (Integrate new solution) Costs of integration of the new solution. Contains costs for developing, software and hardware. Part of the Life-Cost-Model for digital preservation.
CPE (Perform PA) Total costs of performing the Preservation Action over the whole collection. Part of the Life-Cost-Model for digital preservation.
CQA (Quality Assure PA) Total costs of Quality Assurance for Preservation Action over the whole collection. Part of the Life-Cost-Model for digital preservation.
CRM (Record PA Metadata) Total costs of recording metadata about the performed preservation action for the whole collection. Part of the Life-Cost-Model for digital preservation.
PA Total (CIG+CPE+CQA+CRM) Total costs for the Preservation Action. This should be a sum of CIG, CPE, CQA and CRM. Part of the Life-Cost-Model for digital preservation.
Reingest Total costs for the reingest of the objects on which the preservation action has been performed. Part of the Life-Cost-Model for digital preservation.
TCO Total Cost of Ownership. If you use this model for calculating the costs of your preservation action and re-ingest, enter the amount here.
Remarks Remarks about the figures. If you use a costmodel different to TCO or Life, you can enter the data here as well.
Responsible person for execution Enter the person responsible for executing the preservation plan here.
Responsible person for monitoring Enter the person responsible for approving the preservation plan here.
Triggers for re-evaluation Events that trigger a new preservation planning activity (eg. new objects in the collection, a periodic review of the plan, changes in environment or requirements).
The following types of triggers are foreseen:
Periodic ReviewA review scheduled for this plan.
Changed EnvironmentContextual influence factors change that justify a revision of this plan.
Changed ObjectiveThe goals of the institution change and require revising this plan.
Changed Collection ProfileThe profile of a set of objects changes in such a way that the defined measures are not sufficient to keep the objects alive.
